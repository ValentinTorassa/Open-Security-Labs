Cloud se vuelve confuso cuando mezclás conceptos: cuenta, región, servicio, recurso, identidad y permisos. Este lab arma un mapa mínimo sin pedirte publicar nada real.

<Callout type="error">
  No subas account IDs, ARNs reales, nombres de buckets reales, screenshots de
  consola ni salidas de comandos con datos de tu organización. En Open Security
  Labs, cloud se documenta con placeholders y datos ficticios.
</Callout>

## Región y cuenta

Una región es dónde vive el recurso. Una cuenta es el límite administrativo donde se factura, se audita y se aplican permisos.

```text
región de ejemplo: us-east-1
cuenta ficticia:   123456789012
```

Usá `123456789012` como placeholder cuando necesites mostrar forma sin exponer identidad real.

## ARN: la dirección de un recurso

Un ARN identifica recursos de AWS con una forma estable:

```text
arn:aws:iam::123456789012:role/LabReadOnly
 |  |   |       |          |
 |  |   |       |          recurso
 |  |   |       cuenta ficticia
 |  |   región vacía para IAM
 |  servicio
 partición
```

<Callout type="concepto">
  IAM es global, por eso el campo región puede aparecer vacío. Otros servicios,
  como EC2 o Lambda, sí suelen incluir región en el ARN.
</Callout>

## CLI sin filtrar secretos

Podés revisar la instalación y configuración sin mostrar claves.

<TerminalBlock title="vt@labs:~">

```bash
aws --version
aws configure list
```

</TerminalBlock>

Si corrés `aws sts get-caller-identity`, redactá cuenta y ARN antes de guardar evidencia.

```text
{
  "UserId": "AROAXAMPLE:session",
  "Account": "123456789012",
  "Arn": "arn:aws:sts::123456789012:assumed-role/LabReadOnly/session"
}
```

<Callout type="evidencia">
  La evidencia buena no es "mirá mi cuenta real". Es un documento que muestra
  que entendés qué parte identifica la cuenta, qué parte identifica el rol y qué
  parte define el recurso, usando placeholders.
</Callout>
