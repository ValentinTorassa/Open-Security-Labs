Una API no empieza en JWT ni en arquitectura hexagonal. Empieza con una request: método, ruta, headers, body y una respuesta con status, headers y body.

<Callout type="concepto">
  HTTP es un contrato. El cliente expresa una intención y el servidor responde
  qué pasó. Si ese contrato es ambiguo, después autorización, retries y logging
  se vuelven confusos.
</Callout>

## Leé una respuesta completa

`curl -i` muestra headers además del body. Usá una URL pública segura o tu propio servicio local.

<TerminalBlock title="vt@labs:~">

```bash
curl -i https://example.com
```

</TerminalBlock>

La primera línea te dice protocolo y status. Después vienen headers como `Content-Type`, `Cache-Control` o `Server`.

## Método + recurso

Un endpoint debería poder leerse como una frase:

```text
GET    /notes          listar notas
POST   /notes          crear nota
GET    /notes/{id}     ver una nota
PATCH  /notes/{id}     modificar parte de una nota
DELETE /notes/{id}     borrar una nota
```

<Callout type="error">
  Si todo es `POST /do-action`, perdés semántica. Los clientes no saben qué se
  puede cachear, reintentar o auditar. El método importa porque expresa
  intención.
</Callout>

## Status codes como diseño

No uses `200` para todo. Una API más clara separa casos:

```text
caso                         status razonable
creación correcta             201 Created
sin body, operación exitosa   204 No Content
input inválido                400 Bad Request
no autenticado                401 Unauthorized
sin permiso                   403 Forbidden
recurso inexistente           404 Not Found
conflicto de estado           409 Conflict
fallo inesperado              500 Internal Server Error
```

## Content-Type no es decoración

Si mandás JSON, decilo. Si respondés JSON, decilo también.

<TerminalBlock title="vt@labs:~">

```bash
curl -i -X POST https://api.example.test/notes \
  -H "Content-Type: application/json" \
  -d '{"title":"nota de juguete"}'
```

</TerminalBlock>

No hace falta que esa URL exista: la evidencia puede ser el diseño de la request y una matriz de endpoints. Si la corrés contra una API propia, usá datos ficticios.

<Callout type="evidencia">
  Tu entrega debería mostrar una tabla: método, ruta, intención, status feliz,
  errores esperados y ejemplo de body. Eso ya obliga a pensar el contrato antes
  de escribir controladores.
</Callout>
