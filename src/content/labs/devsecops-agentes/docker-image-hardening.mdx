Un contenedor no es una VM mágica. Comparte kernel con el host y tiene permisos. Hardening es reducir qué puede hacer el proceso si algo sale mal.

<Callout type="concepto">
  El objetivo no es hacer el contenedor invulnerable. Es bajar el impacto: menos
  privilegios, menos escritura, menos capacidades y menos secretos expuestos.
</Callout>

## Dockerfile más seguro

Patrón mínimo:

```dockerfile
FROM node:22-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .
RUN addgroup -S app && adduser -S app -G app
USER app
CMD ["node", "server.js"]
```

No alcanza para producción, pero evita correr la app como root dentro del contenedor.

## Runtime con menos permisos

<TerminalBlock title="vt@labs:~">

```bash
docker run --rm \
  --read-only \
  --cap-drop=ALL \
  --tmpfs /tmp \
  app-de-juguete:local
```

</TerminalBlock>

Puede romper apps que escriben en disco o necesitan capabilities. Eso no es malo: te muestra dependencias ocultas.

<Callout type="error">
  No metas secretos en la imagen. Si quedan en una layer, el secreto viaja con
  el artefacto. Usá variables/secret managers del entorno y documentá
  placeholders.
</Callout>

## Checklist de evidencia

```text
control             evidencia
USER no-root         Dockerfile con USER app
capabilities         runbook con --cap-drop=ALL y excepciones justificadas
filesystem           --read-only + tmpfs donde haga falta
secretos             no hay .env ni tokens en layers
docker socket        no montado salvo caso justificado y defensivo
```

<Callout type="evidencia">
  El valor está en explicar tradeoffs. Si `--read-only` rompe logs locales,
  documentá dónde escribe la app y cómo lo resolverías sin abrir todo el FS.
</Callout>
